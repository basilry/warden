# WARDEN P2 도메인 완성 체크리스트

## P2 목표

P2의 목표는 P0/P1에서 만들어진 offline control-plane MVP를 **도메인 규율이 완성된 WARDEN 팀**으로 올리는 것이다.

핵심은 세 가지다.

1. M7 미완료 항목인 **SourceVet Reviewer Agent**를 구현한다.
2. 함수형 policy를 독립된 **Policy Reviewer Agent**로 격상한다.
3. 현재 local capability registry와 이후 실제 stdio MCP를 분리하는 **Real MCP Boundary**를 만든다.

P2는 여전히 기본 실행을 offline/local mode로 유지한다. 실제 외부 MCP나 live LLM 호출은 기본값으로 켜지지 않는다.

## 개발 순서

1. P2.0 SourceVet Reviewer Agent
2. P2.1 Policy Reviewer Agent
3. P2.2 Real MCP Boundary
4. P2.3 P2 regression and demo closure

이 순서를 지켜야 한다. SourceVet이 먼저 들어와야 policy reviewer와 verifier가 검사할 도메인 위험 신호가 늘어난다. MCP boundary는 마지막에 붙여도 local mode가 깨지지 않는다.

### 핵심 타입

```ts
type SourceRiskLevel = "low" | "medium" | "high";

type SourceRiskFlag = {
  id: string;
  level: SourceRiskLevel;
  code:
    | "missing_lineage"
    | "single_source"
    | "circular_lineage"
    | "uncorroborated_upgrade"
    | "stale_report"
    | "unknown_origin";
  summary: string;
  evidenceBundleIds: string[];
};

type SourceReview = {
  status: "pass" | "warn" | "fail";
  reviewedUnitIds: string[];
  riskFlags: SourceRiskFlag[];
  independentCorroborationCount: number;
  sourceLineageSummary: string;
};
```

### 핵심 함수

- `createSourceVetLocalTool(store?: SourceStore): SourceVetLocalTool`
- `registerSourceFromKnowledgeUnit(unit: KnowledgeUnit): SourceProfile`
- `addReportFromClaim(sourceId: string, claim: Claim): SourceReport`
- `assessSource(sourceId: string): SourceAssessment`
- `computeFabricationRisk(profile: SourceProfile, reports: SourceReport[]): SourceRiskLevel`
- `detectCircularLineage(units: KnowledgeUnit[]): SourceRiskFlag[]`
- `detectIndependentCorroboration(units: KnowledgeUnit[]): CorroborationSummary`

- `runSourceVetReviewer(task, context, input): Promise<AgentResult<SourceReview>>`
- `verifySourceRiskFlags(review: SourceReview): VerificationCheck`

### 구현 로직

1. `Evidence Curator`가 만든 `KnowledgeUnit[]`를 SourceVet 입력으로 넘긴다.
2. `registerSourceFromKnowledgeUnit`가 `sourceUri`, `sourceType`, `provenance`, `extractedAt` 기준으로 source profile을 만든다.
3. `addReportFromClaim`가 각 claim을 source report로 변환한다.
4. `detectCircularLineage`가 같은 원문 또는 같은 upstream reference가 반복되는지 검사한다.
5. `detectIndependentCorroboration`가 독립 source type과 독립 source URI 수를 계산한다.
6. 독립검증이 부족한데 reliability가 상향된 evidence는 `uncorroborated_upgrade` flag를 단다.
7. SourceVet 결과는 ACH 자체를 막지는 않는다. 대신 verifier가 source risk를 final brief와 residual risk에 반영했는지 검사한다.
8. `status: fail`인 SourceReview는 Briefing Agent로 바로 넘어가지 않고 verifier fail로 닫는다.

## P2.1 Policy Reviewer Agent

### 목표

현재 `policy.ts`의 함수형 policy를 독립 specialist agent로 격상한다. Supervisor와 tool router 사이에 policy reviewer가 들어가야 “에이전트가 도구를 쓴다”가 아니라 “정책 검토를 통과한 tool plan만 실행된다”는 제품 메시지가 선명해진다.

### 생성 파일

| 파일 | 목적 |
|---|---|
| `src/agent/agents/policy-reviewer.ts` | Policy Reviewer Agent |
| `fixtures/regression/POLICY-002-external-denied-without-approval.json` | 외부 호출 승인 요구 회귀 |
| `fixtures/regression/POLICY-003-policy-change-denied.json` | 정책 변경 차단 회귀 |

### 수정 파일

| 파일 | 수정 내용 |
|---|---|
| `src/agent/policy.ts` | tool call plan review API 추가 |
| `src/agent/team-runner.ts` | WRITE/EXTERNAL/POLICY_CHANGE 실행 전 reviewer 호출 |
| `src/agent/p1-runner.ts` | approval queue와 policy review 연결 |
| `src/agent/mcp-client.ts` | capability invoke 전 policy review hook |
| `src/agent/brief.ts` | policy review summary 추가 |
| `src/agent/regression.ts` | policy reviewer regression 추가 |

### 핵심 타입

```ts
type ToolCallPlan = {
  id: string;
  toolName: string;
  capability: string;
  risk: Risk;
  inputSummary: string;
  requestedBy: AgentRole;
};

type PolicyReviewReport = {
  status: "allow" | "blocked" | "approval_required";
  decisions: PolicyDecision[];
  blockedCallIds: string[];
  approvalCallIds: string[];
  summary: string;
};
```

### 핵심 함수

- `reviewPlannedToolCalls(calls: ToolCallPlan[], context: PolicyContext): PolicyReviewReport`
- `requireApprovalIfExternal(call: ToolCallPlan): PolicyDecision`
- `denyIfPolicyMutation(call: ToolCallPlan): PolicyDecision`
- `denyIfCapabilityMissing(call: ToolCallPlan, registry: CapabilityRegistry): PolicyDecision`
- `renderPolicyReview(report: PolicyReviewReport): string`
- `runPolicyReviewer(task, context, input): Promise<AgentResult<PolicyReviewReport>>`

### 구현 로직

1. specialist agent는 tool을 직접 실행하지 않고 `ToolCallPlan[]`을 만든다.
2. `Policy Reviewer Agent`가 toolName, capability, risk, requestedBy를 평가한다.
3. `READ`는 allow가 기본이다.
4. `WRITE`는 local deterministic tools만 allow하고 trace를 남긴다.
5. `EXTERNAL`은 approval queue에 들어가며 실행은 pending으로 유지한다.
6. `DESTRUCTIVE`와 `POLICY_CHANGE`는 기본 deny 또는 explicit approval required로 둔다.
7. `PolicyReviewReport.status`가 `blocked`이면 해당 specialist task는 failed 또는 blocked로 종료한다.
8. 모든 decision은 trace event와 final brief에 요약된다.

- [x] EXTERNAL 기본 approval required 처리
- [x] POLICY_CHANGE 기본 deny 처리
- [x] capability missing deny 처리
- [x] Policy Reviewer Agent 구현
- [x] team runner에서 tool 실행 전 policy reviewer 호출

- [x] policy review trace event 추가
- [x] final brief에 policy review summary 표시
- [x] POLICY-002 regression 추가
- [x] POLICY-003 regression 추가
- [x] `npm run demo:warden:regression` 통과
- [x] `npm test` 통과

### 완료 기준

- policy decision이 job history, trace, final brief에 모두 나타난다.
- EXTERNAL tool call은 approval pending 없이는 실행되지 않는다.
- 정책 변경 시도는 기본 fail closed로 닫힌다.

## P2.2 Real MCP Boundary

### 목표

P1의 `mcp-client.ts`는 local capability registry다. P2.2에서는 이 계약을 유지하면서 실제 stdio MCP client를 붙일 수 있는 경계를 만든다. 기본 demo는 local mode로 유지한다.

### 생성 파일

| 파일 | 목적 |
|---|---|
| `src/agent/mcp/types.ts` | MCP boundary 공통 타입 |
| `src/agent/mcp/config.ts` | MCP server config parser |
| `src/agent/mcp/local-registry.ts` | 기존 local capability registry 이동 |
| `src/agent/mcp/stdio-client.ts` | stdio MCP client skeleton |
| `src/agent/mcp/router.ts` | local/remote capability router |

### 수정 파일

| 파일 | 수정 내용 |
|---|---|
| `src/agent/mcp-client.ts` | compatibility export 또는 thin wrapper로 축소 |
| `src/agent/p1-runner.ts` | new router 사용 |
| `src/agent/policy.ts` | remote tool risk classification 추가 |
| `src/agent/index.ts` | mcp boundary export 정리 |

### 핵심 함수

- `loadMcpRegistryConfig(path: string): McpRegistryConfig`
- `validateMcpServerConfig(config: unknown): McpRegistryConfig`
- `createLocalCapabilityRegistry(capabilities: Capability[]): CapabilityRegistry`
- `createStdioMcpClient(config: McpServerConfig): McpClient`
- `discoverRemoteTools(server: McpClient): Promise<RemoteToolDescriptor[]>`
- `mapRemoteToolToCapability(tool: RemoteToolDescriptor): Capability`
- `invokeRemoteTool(server: McpClient, toolName: string, input: unknown): Promise<ToolResult>`
- `routeToolCall(plan: ToolCallPlan, router: CapabilityRouter): Promise<ToolResult>`

### 구현 로직

1. 기존 local capability registry를 `mcp/local-registry.ts`로 이동하거나 wrapper로 감싼다.
2. MCP config에는 `mode: "local" | "stdio"`와 `enabled: boolean`을 둔다.
3. `stdio` mode는 config가 없으면 생성 실패한다.
4. remote tool discovery는 allowlist에 없는 tool을 capability registry에 올리지 않는다.
5. remote invoke는 `EXTERNAL` 또는 configured risk로 분류한다.
6. policy reviewer와 approval queue를 통과하지 않으면 remote invoke는 실행되지 않는다.
7. P2에서는 stdio client skeleton과 dry-run invoke까지가 목표다. 실제 외부 server 운영은 P5에서 닫는다.
