# ACH MCP 방산 챌린지 다관점 리뷰

## 결론

제출 방향은 유효하다. 단, 지금 그대로의 `ach-mcp`는 “기술적으로 흥미로운 MCP 프로토타입”에 가깝고, 방산 스타트업 챌린지 제출물로는 **수요기업이 바로 이해할 수 있는 업무 문제**와 **MVP 사용 화면**이 더 필요하다.

가장 좋은 포지션은 “정보기관용”으로 좁히는 것이 아니라, **군·방산기업의 분석·출처검증·사이버 트리아지 업무에서 LLM을 통제 가능하게 쓰는 에이전트 인프라**로 일반화하는 것이다.

## 1. 심사위원 관점

### 강점

- 데모가 이미 돈다. `npm run demo:auto`, SourceVet 자동 데모까지 모델 없이 재현 가능하다.
- “LLM을 신뢰하지 않는 방식으로 쓴다”는 명제가 명확하다.
- 방산/공공 부문에서 중요한 키워드인 출처추적, 감사가능성, 폐쇄망, HITL, 최소권한과 잘 맞는다.
- 단순 챗봇이 아니라 MCP 기반 툴 레이어라 기술 차별성이 있다.

### 약점

- 현재 README와 데모는 정보분석 tradecraft에 익숙한 사람에게는 강하지만, 일반 창업 심사위원에게는 다소 추상적일 수 있다.
- 방산 수요기업이 “우리 업무에서 어디에 붙지?”라고 물으면 아직 답이 넓다.
- 사업화 모델이 “도구/프레임워크”로 남아 있어 구매자, 도입 단위, PoC 성공지표를 더 선명하게 해야 한다.

### 보완

- 제출 문구는 “AI 분석관”보다 “검증 가능한 AI 분석 보조/의사결정 보조”로 낮춰 잡는다.
- 사용 시나리오를 3개로 제한한다.
  - 방산 공급망 이상징후 분석
  - 공개출처 기반 위협/동향 분석
  - 협력사·출처·보고 신뢰도 검증
- 심사용 화면은 “분석 매트릭스, 출처 신뢰도, 감사 로그, 승인 대기” 4개 탭이면 충분하다.

## 2. 군·수요기업 관점

### 기대 가치

- 빠른 1차 판단보다 “왜 그렇게 판단했는가”를 남기는 것이 가치다.
- 단일 출처 과신, 보고 편향, 경보 피로를 줄이는 데 설득력이 있다.
- 폐쇄망/온프렘/로컬 LLM 전제가 방산 환경과 맞는다.

### 우려

- 실제 내부 데이터와 연결하기 전에는 데모가 장난감처럼 보일 수 있다.
- “AI가 판단한다”는 인상을 주면 도입 저항이 생길 수 있다.
- 수요기업마다 도메인 용어, 문서 양식, 승인 체계가 달라 커스터마이징 비용이 생긴다.

### 보완

- MVP는 실제 데이터 연결을 약속하기보다 **합성 데이터 + 가져오기 어댑터 구조**를 보여준다.
- 판단권은 사람에게 있다는 문구를 반복한다.
- 데이터 커넥터는 후순위로 두고, 먼저 “감사 가능한 분석 절차”를 제품화한다.

## 3. 기술 관점

### 강점

- ACH와 SourceVet의 판단 로직이 결정적 코드로 분리되어 있어 테스트 가능하다.
- MCP Tools/Resources 구조가 명확하다.
- append-only audit, watcher notification, 신뢰도 등급 같은 규율 프리미티브가 이미 들어 있다.
- 의존성이 작고 TypeScript 단일 프로젝트라 패키징 난도가 낮다.

### 약점

- 현재는 “MCP 서버”이지 “사용자가 쓰는 앱”은 아니다.
- 통제형 에이전트 런타임은 아직 설계 문서 단계다.
- 저장소가 JSON 파일 중심이라 멀티사용자/권한/운영 환경에는 부족하다.
- 테스트 파일이 별도로 보이지 않고 데모가 사실상 회귀검증 역할을 한다.

### 보완

- P0에서 얇은 에이전트 런타임을 만든다.
  - 모델 어댑터는 mock 우선
  - MCP 클라이언트 매니저
  - 정책 엔진
  - HITL 승인
  - 감사 로그
- P1에서 SQLite 또는 Postgres 저장소, 구조화 로그, 컨테이너 패키징을 붙인다.

## 4. 보안·방산 적합성 관점

### 강점

- stdio 기반 MCP는 네트워크 노출을 줄일 수 있다.
- 로컬 모델/에어갭/egress 0 스토리와 잘 맞는다.
- 프롬프트 인젝션 대응을 정책 계층에서 설명할 수 있다.
- 판단 결과가 LLM 내부 추론이 아니라 코드 산출물이므로 감사와 재현성이 좋다.

### 위험

- “정보기관/방첩” 표현이 외부 제출에서 과하게 보일 수 있다.
- 외부 공개출처 수집을 붙이면 법무·약관·보안 이슈가 생긴다.
- 에이전트가 외부 호출을 하게 되는 순간 방산 환경과 충돌할 수 있다.

### 보완

- 제출서에는 “실작전/실데이터 무관, 합성 데이터 기반”을 명시한다.
- 외부 수집은 P2로 미루고, P0/P1은 로컬 파일/사전 적재 데이터만 쓴다.
- “분석 자동화”가 아니라 “분석 절차 통제·검증 자동화”로 표현한다.

## 5. 사업성 관점

### 좋은 신호

- 방산 AI 도입의 핵심 장애물인 신뢰, 감사, 보안, 망분리를 정면으로 다룬다.
- 수요기업 과제 모집공고가 먼저 열린 것은 실제 과제 수요를 모으는 단계로 볼 수 있다.
- 스타트업 챌린지에서는 “수요기업 과제에 맞춰 MVP를 조정”하는 전략이 가능하다.

### 약한 부분

- 구매자가 누구인지 아직 넓다.
- 가격/도입 모델이 없다.
- PoC 성공지표가 기술 중심이다.

### 보완

- 1차 고객: 방산기업 전략/기술정보/보안/사업개발 부서.
- 2차 고객: 군·공공 안보 분석 조직.
- PoC 성공지표:
  - 분석 케이스 처리 시간 감소
  - 출처 미기재 주장 차단율
  - 감사 로그 재현 가능성
  - 판단 변경 알림의 소음 감소

## 6. 사용자 경험 관점

### 현재 문제

CLI 데모는 강하지만, 심사위원이나 수요기업 담당자가 직접 이해하기에는 진입장벽이 있다.

### 필요한 화면

- Case Console: 질문, 가설, 증거, 판정 매트릭스
- SourceVet Console: 출처, 보고 이력, 날조위험, 플래그
- Audit Timeline: 누가 언제 무엇을 호출했고 왜 거부/승인됐는지
- HITL Queue: 승인 대기 툴콜, 위험등급, 승인/거부

## 7. 최종 판단

현재 프로젝트는 “기술 원석”으로는 충분히 좋다. 제출용으로는 다음 한 문장으로 정리해야 한다.

> 방산 조직이 LLM을 바로 믿고 쓰지 않도록, 분석 절차·출처검증·승인·감사를 MCP와 결정적 코드로 강제하는 통제형 AI 에이전트 MVP.

## 우선순위

1. 방산 챌린지용 문제정의 1페이지 작성.
2. MVP 화면 또는 녹화 가능한 데모 UI 제작.
3. WARDEN 통제형 에이전트 런타임 P0 구현.
4. 공고 후속 스타트업 모집 일정 모니터링.
5. 수요기업 과제 공개 시 키워드 매칭.
