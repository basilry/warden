# WARDEN 현재 개발 현황 및 다관점 리뷰

## 결론

현재 WARDEN은 **로컬 실행 가능한 에이전트 런타임 알파** 수준까지 올라왔다.

초기 상태의 "하네스/스캐폴드"를 지나, 지금은 `warden` CLI에서 자연어 목표를 넣으면 다음 흐름이 실제로 돈다.

1. 목표 수신
2. LLM 계획 제안
3. 정책/권한 계층에서 도구 실행 여부 결정
4. 내부 WARDEN 팀 분석 실행
5. 외부 OSINT 필요 시 승인 대기
6. 승인 후 라이브 OSINT 수집
7. SourceVet으로 출처 검토
8. ACH 재평가
9. 답변 델타 생성
10. 런타임 상태 저장/복원

다만 아직은 **운영형 프로덕션 에이전트**라기보다는 **통제된 로컬 알파**다. 핵심 기능은 돌아가지만, 외부 검색 품질, 승인 재개 실패 처리, 장기 저장소, MCP 경계의 일관성, 브라우저/웹 스크래핑 수집기는 다음 단계에서 보강해야 한다.

## 다관점 리뷰

### 제품/사용자 경험: 78 / 100

좋은 점:

- 사용자는 이제 `warden` 명령어로 에이전트를 실행할 수 있다.
- CLI 로그가 한국어로 바뀌었고, 진행 단계가 보인다.
- 실행 결과가 단순 run id가 아니라 답변 형태에 가까워졌다.

부족한 점:

- 장시간 대기 중 내부 진행률이 아직 충분히 풍부하지 않다.
- 승인 후 재개 결과가 대화형으로 자연스럽게 이어지는 경험은 더 다듬어야 한다.
- 최종 답변의 깊이는 수집된 live evidence 품질에 크게 좌우된다.

### 에이전트 아키텍처: 82 / 100

좋은 점:

- LLM 제안은 proposal-only로 제한되어 있고, 정책/MCP/승인 계층이 권위를 가진다.
- 팀 실행, ACH, SourceVet, OSINT, runtime persistence가 모듈화되어 있다.
- 하위 에이전트가 무분별하게 늘어나는 구조에서 벗어나 필요한 역할 위주로 정리되었다.

부족한 점:

- 일부 경로에서는 MCP server가 아닌 내부 connector를 직접 호출한다.
- planner가 더 풍부한 multi-step 작업을 안정적으로 수행하려면 tool result memory와 retry strategy가 필요하다.

### 보안/통제/OPSEC: 83 / 100

좋은 점:

- 외부 호출은 기본 차단된다.
- live OSINT는 opt-in과 approval을 모두 요구한다.
- 허용되지 않은 외부 source는 fail-closed된다.

부족한 점:

- 승인된 resume이 실패했을 때 상태 전이가 더 명확해야 한다.
- 외부 provider별 rate-limit/cooldown이 없다.
- 감사 로그는 있으나 장기 감사/검색용 저장소는 아직 MVP다.

### MCP 경계: 78 / 100

좋은 점:

- ACH MCP는 실제 stdio server/client 구조로 분리되었다.
- OSINT MCP도 `search_news` 도구로 구성되었다.

부족한 점:

- runtime resume은 현재 OSINT MCP client를 항상 통과하지 않고 내부 live OSINT connector를 호출한다.
- "분석 코어는 MCP만 통해 외부 기능에 접근한다"는 원칙을 강제하려면 runtime 경계를 더 엄격하게 바꿔야 한다.

### OSINT 수집: 70 / 100

좋은 점:

- 하드코딩된 fixtures만 쓰던 상태에서 자연어 검색 기반 수집 경로가 생겼다.
- GDELT, Brave, Investing.com 스코프 검색, RSS 조합이 가능하다.
- 실패 시 차단 결과와 warning으로 내려온다.

부족한 점:

- GDELT는 실제 스모크에서 timeout/429가 있었다.
- Brave는 API 키 없이는 동작하지 않는다.
- web scraping 기반 fallback은 아직 본격 구현되지 않았다.
- 검색 결과의 품질 랭킹, 중복 제거, 출처 신뢰도 profile이 더 필요하다.

### 분석 신뢰성: 80 / 100

좋은 점:

- SourceVet 통과 evidence만 승격하는 구조가 있다.
- ACH rerun으로 외부 근거 반영 전후의 가설 변화를 볼 수 있다.

### 코드 품질/유지보수: 78 / 100

## 코드리뷰 주요 지적

### P1. 승인 재개 실패 시 런타임 상태가 어정쩡해질 수 있음

위치:

- `/Users/basilry/Projects/02021_warden_agents/src/runtime/loop.ts`
- `/Users/basilry/Projects/02021_warden_agents/src/runtime/resume.ts`

문제:

`approvePendingApproval` 이후 live OSINT fetch나 resume 단계에서 예외가 나면, approval은 이미 처리되었는데 run 상태는 `running` 또는 부분 갱신 상태로 남을 수 있다.

영향:

- 사용자는 승인했는데 최종 답변이 나오지 않을 수 있다.
- 재시도 UX가 애매해진다.
- 감사 로그상 approval은 resolved인데 실제 run completion은 실패한 상태가 될 수 있다.

권장 수정:

- resume 전체를 try/catch로 감싼다.
- 실패 시 `run.resume_failed` 이벤트를 남긴다.
- 상태를 `failed` 또는 `waiting_approval_retryable`로 명확히 둔다.
- 기존 답변/ACH 결과는 보존하고, 실패 원인을 사용자에게 표시한다.

### P1. OSINT 검색 source 선택과 static allowlist 의미가 섞여 있음

위치:

- `/Users/basilry/Projects/02021_warden_agents/src/runtime/live-osint-fetch.ts`

문제:

## 현재 상태 판단

현재 WARDEN은 더 이상 "정적 HTML 리포트 생성기"나 "데모용 스캐폴드"가 아니다. CLI에서 실행되는 에이전트 런타임이고, 정책/승인/MCP/ACH/SourceVet/OSINT가 실제로 연결되어 있다.

그러나 사용자가 기대하는 수준, 즉 "자연어로 물으면 인터넷과 MCP를 활용해서 충분히 좋은 실시간 분석 답변을 주는 WARDEN"까지는 아직 한 단계가 남아 있다.

가장 중요한 다음 작업은 다음 3개다.

1. 승인 재개 실패 처리 보강
2. OSINT 수집 품질 계층 강화
3. 웹 스크래핑 MCP 추가 및 MCP 경계 정리

이 3개가 끝나면 WARDEN은 로컬 알파에서 **실사용 베타**에 가까워질 수 있다.
