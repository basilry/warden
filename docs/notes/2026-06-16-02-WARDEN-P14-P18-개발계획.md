# WARDEN P14-P18 개발 계획

## 목적

이 문서는 [[2026-06-16-01-WARDEN-현재개발현황-다관점-리뷰]]에서 도출된 후속 과제를 실제 개발 가능한 단위로 쪼갠 계획서다.

현재 WARDEN은 로컬 실행 가능한 에이전트 런타임 알파다. 다음 목표는 다음과 같다.

1. 자연어 질의에서 실제 웹/뉴스 근거 수집 능력 강화
2. 승인 이후 실패해도 런타임 상태가 망가지지 않도록 보강
3. OSINT provider 품질과 rate-limit 대응 개선
4. 외부 수집/분석 기능의 MCP 경계 일관화
5. CLI operator UX 개선

## 병렬 개발 전략

서로 충돌을 줄이기 위해 write set을 다음처럼 나눈다.

| 트랙 | 담당 범위 | 주요 파일 |
|---|---|---|
| P14 Web Scraper MCP | HTML fetch/extract, MCP tool, scrape regression | `src/connectors/osint/html-scraper.ts`, `src/mcp/osint/*`, `demo/run-warden-osint-scrape-mcp-regression.ts` |
| P15 Runtime Resume Hardening | 승인 재개 실패 상태/이벤트/테스트 | `src/runtime/loop.ts`, `src/runtime/types.ts`, `demo/run-warden-resume-failure-regression.ts` |
| P16 OSINT Quality Layer | provider telemetry, reliability profile, error kind | `src/connectors/osint/provider-quality.ts`, `src/connectors/osint/search.ts`, `demo/run-warden-osint-provider-quality-regression.ts` |
| P17 MCP Boundary Cleanup | resume에서 OSINT MCP invoker 우선 경로 | `src/runtime/resume.ts`, `src/agent/mcp/osint-client.ts`, `demo/run-warden-osint-mcp-boundary-regression.ts` |
| P18 Operator UX | CLI 진행 표시/inspect/approval UX | `src/cli/warden.ts`, `src/runtime/types.ts`, `docs/runtime.md` |

## P14. Web Scraper MCP

### 목표

검색 API가 실패하거나 결과가 부족할 때, 승인된 URL을 fetch해서 HTML 본문을 추출하고 SourceVet/ACH에 연결 가능한 evidence 후보로 만든다.

초기 구현은 브라우저 자동화가 아니라 HTTP fetch 기반으로 제한한다.

### 파일/구조

새 파일:

- `src/connectors/osint/scrape-types.ts`
- `src/connectors/osint/html-scraper.ts`
- `demo/run-warden-osint-scrape-mcp-regression.ts`

수정 파일:

- `src/mcp/osint/types.ts`
- `src/mcp/osint/tools.ts`
- `fixtures/mcp/osint-server.json`
- `package.json`

### 주요 타입

```ts
export type OsintScrapeRequest = {
  urls: string[];
  runId: string;
  approvalId: string;
  maxChars?: number;
  maxLinks?: number;
  timeoutMs: number;
  userAgent: string;
};

export type OsintScrapedDocument = {
  url: string;
  canonicalUrl?: string;
  title: string;
  excerpt: string;
  links: { text: string; href: string }[];
  capturedAt: string;
};
```

### 주요 함수

- `runApprovedHtmlScrape(request, options)`
- `extractHtmlDocument(html, sourceUrl, limits)`
- `normalizeScrapedDocumentsToKnowledgeUnits(...)`
- `validateScrapeUrl(url)`

### 체크리스트

- [x] HTTP/HTTPS URL만 허용
- [x] `WARDEN_OSINT_LIVE_OPT_IN=true` 없으면 MCP tool 차단
- [x] `maxChars`, `maxLinks`, `maxUrls` 제한
- [x] `<title>`, canonical link, 본문 텍스트 추출
- [x] script/style/noscript/svg 등 노이즈 제거
- [x] 상대 링크를 절대 URL로 정규화
- [x] localhost/private IP/link-local/`.local` URL 차단
- [x] mock fetch 기반 regression
- [x] OSINT MCP에 `scrape_news` 도구 등록
- [x] README/env/MCP 문서 업데이트

### 완료 기준

- `npm run demo:warden:osint-scrape-mcp` 통과
- `npm test` 통과
- live opt-in이 꺼진 상태에서 scrape tool이 fail-closed

## P15. Runtime Resume Hardening

### 목표

사용자가 외부 OSINT를 승인한 뒤 fetch/SourceVet/ACH 재평가가 실패해도 런타임 상태가 애매하게 남지 않도록 한다.

### 파일/구조

수정 파일:

- `src/runtime/loop.ts`
- `src/runtime/types.ts`

새 테스트:

- `demo/run-warden-resume-failure-regression.ts`

### 주요 함수

- `approveRuntimeApproval(...)`
- `markRuntimeResumeFailed(run, approval, error)`
- `emit(..., "run.resume_failed", ...)`

### 로직

현재 흐름:

1. approval resolved
2. run status `running`
3. resume 실행
4. resume 실패 시 예외 전파 가능

개선 흐름:

1. approval resolved
2. run status `running`
3. resume 실행
4. 실패 시 run status `failed`
5. `completedAt`, `updatedAt`, `error` 기록
6. `run.resume_failed` 이벤트 기록
7. 기존 answer/ACH/fetchedEvidence는 보존
8. repository에 실패 상태 저장

### 체크리스트

- [x] `RuntimeEventType`에 `run.resume_failed` 추가
- [x] resume 실패 helper 추가
- [x] `approveRuntimeApproval`에서 resume try/catch 처리
- [x] 실패 상태 저장 보장
- [x] 실패 이벤트 emit
- [x] 기존 answer 보존 검증
- [x] approval은 resolved 상태로 남는지 검증
- [x] mock fetch failure regression

### 완료 기준

- 승인 이후 live OSINT fetch 실패 시 `approveRuntimeApproval`이 throw하지 않고 failed run을 반환
- 반환 run에 `error`와 `run.resume_failed` 이벤트가 존재
- 기존 `demo:warden:live-osint-resume` 통과

## P16. OSINT Provider Quality Layer

### 목표

검색 provider별 latency, 실패 종류, rate-limit, 기본 신뢰도 profile을 기록하고 검색 품질을 개선한다.

### 파일/구조

새 파일:

- `src/connectors/osint/provider-quality.ts`
- `demo/run-warden-osint-provider-quality-regression.ts`

수정 파일:

- `src/connectors/osint/search.ts`
- `src/connectors/osint/search-types.ts`
- `fixtures/osint/search-sources.json`

### 주요 타입

```ts
export type OsintProviderTelemetry = {
  sourceId: string;
  status: "succeeded" | "failed" | "skipped";
  latencyMs: number;
  errorKind?: "timeout" | "rate_limited" | "http_error" | "config_invalid" | "parse_error";
  message?: string;
};
```

### 주요 함수

- `classifyProviderError(error)`
- `createProviderTelemetry(source, startedAt, status, error?)`
- `readSourceReliability(source, fallback)`

### 체크리스트

- [x] 429를 `rate_limited`로 분류
- [x] timeout을 `timeout`으로 분류
- [x] HTTP error를 `http_error`로 분류
- [x] provider별 latency 기록
- [x] source reliability profile이 있으면 문서 reliability에 반영
- [x] `search-sources.json` parser가 reliability/cooldown/backoff 필드를 보존
- [x] warnings에 raw error만 넣지 않고 분류된 메시지 포함
- [x] mock fetch regression

### 완료 기준

- provider telemetry가 search result에 포함
- 기존 search MCP regression 통과
- source reliability profile이 실제 KnowledgeUnit reliability로 반영

## P17. MCP Boundary Cleanup

### 목표

외부 수집과 분석 기능이 MCP 경계를 일관되게 통과하도록 정리한다. 우선은 큰 refactor 대신 runtime resume에서 OSINT MCP invoker를 주입할 수 있는 경계를 만든다.

### 파일/구조

수정 파일:

- `src/runtime/resume.ts`
- `src/runtime/loop.ts`
- `src/agent/mcp/osint-client.ts`

새 테스트:

- `demo/run-warden-osint-mcp-boundary-regression.ts`

### 주요 타입

```ts
export type OsintSearchInvoker = (input: {
  query: string;
  runId: string;
  approvalId: string;
  maxResults: number;
}) => Promise<OsintFetchResult>;
```

### 로직

1. `ResumeApprovedExternalFetchOptions`에 `osintSearchInvoker` 또는 `osintMcpInvoker` 추가
2. live opt-in + invoker 존재 시 invoker 우선 사용
3. invoker가 없으면 기존 `runApprovedLiveOsintFetch` fallback
4. regression에서 mock invoker가 호출되는지 검증

### 체크리스트

- [x] resume option 추가
- [x] invoker 우선 경로 구현
- [x] runtime dependency에서 invoker 전달 가능
- [x] fallback 유지
- [x] mock invoker regression
- [x] OSINT MCP client helper 검토

### 완료 기준

- MCP invoker 주입 시 runtime resume이 connector를 직접 호출하지 않음
- 기존 live OSINT resume regression 통과
