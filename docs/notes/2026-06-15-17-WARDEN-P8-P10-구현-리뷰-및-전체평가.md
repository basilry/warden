# WARDEN P8-P10 구현 리뷰 및 전체 프로젝트 평가

## 구현 요약

- P8: planner proposal을 schema/allowlist/capability/risk 검증 후에만 `RoutedToolCall` 후보로 승격한다.
- P9: CLI/API approval approve/reject와 deterministic local fetch resume 경로를 추가했다.
- P10: 한국/동북아 공급망 domain classifier, profile, retrieval, answer grounding을 추가했다.
- README와 runtime docs에 실제 실행/승인 재개 방식을 반영했다.

- `npm run demo:warden:approval-resume`: 통과

제한: 로컬 HTTP 서버 회귀는 `127.0.0.1` 포트 바인딩이 필요하므로 restricted sandbox에서는 권한 있는 실행이 필요했다.

## 코드리뷰 Findings

### P1: 승인 후 evidence는 answer에만 병합되고 ACH/SourceVet 재평가는 아직 수행하지 않는다

- 위치: `src/runtime/loop.ts` `completeApprovedExternalFetch()`
- 현재 동작: `previousAnswer`를 유지하고 fetched evidence, uncertainty, authority refs만 갱신한다.
- 영향: 승인 후 새 근거가 들어와도 ACH survivor ranking은 바뀌지 않는다. 즉 “근거가 추가된 답변”이지 “근거 기반 재분석”은 아니다.
- 다음 조치: `resumeStage=sourcevet|ach`를 도입해 fetched `KnowledgeUnit[]`을 SourceVet reviewer와 ACH analyst 입력으로 재투입한다.

### P1: P10 retrieved evidence는 runtime answer에는 반영되지만 team evidence matrix에는 들어가지 않는다

- 위치: `src/runtime/loop.ts` `attachDomainGrounding()`, `src/agent/agents/evidence-curator.ts`
- 현재 동작: domain grounding은 `RuntimeRun.outputs.domainGrounding`에 저장되고 answer composer가 사용한다. Evidence Curator는 여전히 `createSupplyChainKnowledgeFixture()`만 사용한다.
- 영향: 넓은 공급망 질문의 답변은 domain 근거를 보여주지만, ACH 가설과 matrix는 기존 fixture case frame에 묶인다.
- 다음 조치: `RunOptions.domainGrounding` 또는 team input에 retrieved units를 전달하고 Evidence Curator가 fixture와 병합하게 한다.

### P2: external OSINT fetch는 아직 live connector가 아니라 deterministic fixture이다

- 위치: `src/runtime/external-fetch.ts`
- 현재 동작: approval이 있어도 실제 네트워크 호출 없이 `fixture://warden/approved-osint/*` KnowledgeUnit을 반환한다.
- 영향: 현재 정보성 답변에는 유용하지만 최신 OSINT나 운영 판단에는 부족하다.
- 다음 조치: live connector는 allowlist, redaction, SourceVet, audit policy가 준비된 뒤 붙인다.

### P2: CLI approval state는 대화형 프로세스 내부에만 존재한다

- 위치: `src/cli/warden.ts`
- 현재 동작: `warden` chat mode는 process-local `RuntimeState`를 사용한다. `warden run`은 1회 실행 후 종료되므로 이후 `/approve`로 이어갈 수 없다.
- 영향: 재개 가능한 운영은 `warden` chat mode 또는 `warden server`를 써야 한다.
- 다음 조치: durable runtime state가 필요하면 JSONL/SQLite storage와 runtime server state를 연결한다.

## 다관점 평가

### 사용자 가치

- 이전의 “실행했는데 아무것도 답하지 않는” 상태에서 벗어나, CLI에서 답변/근거/한계/승인 필요/다음 단계를 볼 수 있다.
- 승인 대기에서 끝나는 이유와 승인 후 재개 방법이 CLI/API/README에 드러난다.

### 보안/통제

- 모델 proposal은 직접 실행되지 않고 schema, allowlist, policy, MCP router를 통과한다.
- external action은 human approval 전까지 실행되지 않는다.
- 승인 후 fetch도 현재는 local fixture라 egress risk가 없다.

### 분석 품질

- ACH, verifier, answer composer, domain grounding이 결합되어 “그럴듯한 일반 챗봇 답변”보다 근거와 한계를 더 잘 드러낸다.
- 다만 ACH의 입력과 P10 retrieval이 아직 완전히 결합되지 않아, 분석 품질은 “runtime briefing 수준”이지 “운영 분석 자동화” 수준은 아니다.

### 운영성

- `warden`, `warden run`, `warden server`, `/approve`, HTTP approve endpoint가 있어 기본 운영 흐름은 생겼다.
- durable state, concurrency, retry, authn/authz는 아직 production 수준이 아니다.

## 현재 수준 판정

현재 WARDEN은 **offline-first controlled agent runtime MVP+** 수준이다.

- CLI와 HTTP server가 실제로 실행된다.
- LLM/Codex는 proposal 역할로 제한된다.
- ACH/approval/domain grounding/answer 출력이 연결되어 demo와 PoC 설명이 가능하다.
- 실제 최신 OSINT, 조직 인증, durable runtime resume, ACH 재평가 루프까지는 다음 phase가 필요하다.
