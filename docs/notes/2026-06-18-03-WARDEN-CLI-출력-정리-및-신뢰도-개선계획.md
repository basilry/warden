# WARDEN CLI 출력 정리 및 신뢰도 개선계획

## 1. 작성 배경

2026-06-18 실사용 로그 기준으로 WARDEN은 자연어 질문, Codex model proposal, 분석계획, 온톨로지 확장, 로컬 RAG, ACH, SourceVet, forecast, approval resume까지 실제로 동작한다.

그러나 현재 출력은 다음 문제가 있다.

1. 같은 판단이 `답변`, `보안분석 보고서`, `상태 요약`, `승인 후 재개 결과`에서 반복된다.
2. 로그가 단계별로 분리되어 보이지 않아 사용자는 "무슨 단계가 끝났고 지금 어디인지"를 즉시 파악하기 어렵다.
3. 실시간 OSINT 수집 후 기사 전문 또는 boilerplate가 그대로 `근거`에 출력된다.
4. 최종 결론이 "현재 생존 가설" 중심이라 사용자 입장에서는 어떤 판단이 내려졌는지 덜 확정적으로 보인다.
5. 신뢰도 `낮음`의 원인이 보이기는 하지만, 무엇을 하면 `중간/높음`으로 올라가는지 운영 기준이 부족하다.

핵심 방향은 **WARDEN이 내부적으로는 full evidence를 보존하되, 사용자 CLI에는 decision-oriented compact report만 보여주는 것**이다.

## 2. 목표 UX

WARDEN 실행 결과는 다음 형태로 보여야 한다.

```text
────────────────────────────────────────
1. 분석계획
────────────────────────────────────────
도메인: 안보
시나리오: 일반 안보
검색계획: 3개

────────────────────────────────────────
2. 근거수집
────────────────────────────────────────
로컬 RAG: 4건
실시간 OSINT: 5건 수집, 2건 채택, 3건 제외

────────────────────────────────────────
3. 검증
────────────────────────────────────────
SourceVet: 통과
ACH: 생존 가설 1개
Forecast: 15.17%, 신뢰도 낮음

────────────────────────────────────────
4. 결론
────────────────────────────────────────
현재 판단: 판정 보류
이유: 질문과 직접 연결되는 미국 정부/주요 언론 근거가 부족함.
다음 행동: 관련 공식 반응/전문 분석 출처를 추가 수집해야 함.

────────────────────────────────────────
5. 참조한 자료
────────────────────────────────────────
- 연합뉴스, "기사 제목", 2026-06-18, https://...
- Reuters, "기사 제목", 2026-06-18, https://...
```

중요한 점은 사용자가 보고 싶은 것은 내부 trace 전체가 아니라 **결론, 근거 출처, 불확실성, 다음 행동**이다.

## 3. 문제 진단

### 3.1 중복 출력

현재 흐름:

1. `printRuntimeEvent`가 loop/model/tool/approval 이벤트를 출력한다.
2. `printRunAnswer`가 direct answer, key findings, evidence, uncertainty, next steps를 출력한다.
3. `printSecurityReport`가 같은 key findings와 forecast/uncertainty/gap/watch indicators를 다시 출력한다.
4. `printRunResult`가 status, survivor, analysis diagnostics를 또 출력한다.
5. approval 후 resume되면 다시 2~4를 전체 출력한다.

결과적으로 같은 내용이 최소 2~3회 반복된다.

### 3.2 단계 구분 부족

이벤트는 `[모델]`, `[분석계획]`, `[도구]`처럼 prefix가 붙지만, 사용자는 전체 실행 흐름을 "단계"로 보지 못한다.

필요한 구분:

- 준비: objective, config, model, iterations
- 계획: LLM proposal, deterministic fallback, search plan
- 수집: local RAG, OSINT, MCP fetch
- 분석: ACH, claim graph, evidence ledger, forecast
- 검증: SourceVet, policy, approval
- 결론: verdict, confidence, next action

### 3.3 기사 전문 출력

현재 `KnowledgeUnit.claims[0].text`에 HTML article body 또는 boilerplate가 들어오면 CLI `근거` 섹션에 그대로 출력된다.

문제:

- 터미널 가독성이 급격히 나빠진다.
- 저작권/재배포 리스크가 생긴다.
- 핵심 질문과 무관한 기사 전문이 나오면 WARDEN 신뢰도가 떨어진다.
- 분석 내부 참조와 사용자 표시가 분리되어 있지 않다.

정책:

- 내부 저장: 원문, redacted artifact, extracted text 유지.
- 사용자 출력: 제목, URL, 출처, 게시일, 신뢰도, 1문장 요지만 출력.
- 전문 출력은 `--verbose` 또는 `--debug-evidence`에서만 허용한다.

### 3.4 결론이 덜 확정적으로 보임

현재 문장:

> 현재 통제 분석에서는 X 가설이 생존했습니다.

이는 ACH 내부 관점으로는 정확하지만, 사용자가 원하는 것은 다음 중 하나다.

- 결론 가능: "현재 판단은 A다."
- 잠정 판단: "A가 가장 그럴듯하지만 추가 근거가 필요하다."
- 판정 보류: "직접 근거가 부족해 결론을 내리면 안 된다."
- 승인 필요: "외부 수집 승인 없이는 판단 불가."

따라서 `survivor`를 사용자 결론으로 직접 노출하지 말고, 별도의 `RuntimeVerdict`를 만들어야 한다.

## 4. 개발 목표

### 목표 A. 출력 중복 제거

- `답변`과 `보안분석 보고서`를 동시에 풀 출력하지 않는다.
- 기본 출력은 단일 compact report로 통합한다.
- 상세 분석 보고서는 `--verbose`, `/details`, `WARDEN_LOG_LEVEL=verbose`에서만 출력한다.

### 목표 B. 단계별 구분선과 타이틀 추가

- 각 주요 단계 시작/종료에 divider를 출력한다.
- 이벤트 prefix는 유지하되, stage header 아래로 묶는다.
- approval 전/후를 명확히 분리한다.

### 목표 C. 기사 전문 미출력

- HTML/뉴스 source는 `EvidenceCitation`으로 표시한다.
- article body는 artifact/store에만 저장하고 사용자 CLI에는 title/url만 표시한다.
- 추출 실패 시에도 raw text를 그대로 출력하지 않는다.

### 목표 D. 결론 패널 도입

- `RuntimeVerdict`를 생성한다.
- report 맨 위에 `현재 판단`, `판정 수준`, `신뢰도`, `핵심 이유`, `다음 행동`을 표시한다.
- low confidence일 때도 "불확실하지만 가장 가능성이 높은 판단"과 "판정 보류"를 구분한다.

### 목표 E. 신뢰도 개선 공정 문서화 및 구현

- low confidence 원인을 구조화한다.
- 신뢰도를 올리기 위한 수집/검증/분석 조건을 정의한다.
- regression fixture로 신뢰도 상승/하락 조건을 잠근다.

## P31. Source Relevance Gate

목표: 질문과 무관한 뉴스가 ACH/답변 신뢰도를 올리지 못하게 한다.

현재 사례:

- 질문: 두만강 공동개발 사건에 대한 미국의 반응
- 수집 기사: 오바마 센터, 미국 건국 250주년, 에볼라 보건외교 등
- 문제: 기사 자체가 실시간 뉴스이긴 하지만 질문과 직접 관련이 낮다.

대상 파일:

- `src/connectors/osint/search.ts`
- `src/connectors/osint/discovery.ts`
- `src/runtime/live-osint-fetch.ts`
- `src/runtime/resume-evidence.ts`
- 신규 `src/runtime/source-relevance.ts`

구현 항목:

- [x] objective entity extraction: 국가, 지역, 사건명, 행위자, 날짜 (현재 keyword/entity term 기반)
- [x] source title/body entity match score
- [x] query term coverage score
- [ ] recency score
- [x] source authority score
- [ ] language match score
- [x] relevance threshold 미달 source는 `rejectedUnits`로 이동
- [x] rejected source는 기본 출력에서 `제외된 자료 N건`으로만 표시
- [x] ACH promotion은 relevance threshold 이상만 허용

권장 threshold:

- `0.75+`: 직접 근거
- `0.55~0.74`: 배경 근거
- `0.35~0.54`: 참고 가능하지만 판단 근거 불가
- `<0.35`: 제외

회귀 기준:

- [x] 질문과 무관한 기사 5건은 SourceVet pass여도 ACH support를 올리지 않음 (`demo/run-warden-source-relevance-regression.ts`)
- [x] 질문 관련 공식/전문 출처 2건 이상이면 support 상승
- [x] low relevance source는 "참조했지만 판단 근거 제외"로 분류

## P32. Runtime Verdict Engine

목표: ACH survivor를 사용자 결론으로 그대로 노출하지 않고, 결론용 verdict를 별도 생성한다.

대상 파일:

- 신규 `src/runtime/verdict.ts`
- `src/runtime/answer.ts`
- `src/runtime/report-composer.ts`
- `src/runtime/types.ts`

데이터 구조:

```ts
export type RuntimeVerdict = {
  status: "blocked" | "insufficient_evidence" | "provisional" | "supported" | "strong";
  decision: string;
  topHypothesis?: string;
  confidence: "low" | "medium" | "high";
  confidenceScore: number;
  reasons: string[];
  blockers: string[];
  nextAction: string;
};
```

판정 규칙:

- [x] 승인 대기 + 외부 근거 없음 → `blocked`
- [x] 직접 관련 source 0~1건 → `insufficient_evidence`
- [x] 관련 source 2건 이상 + SourceVet pass + ACH survivor 1개 → `provisional`
- [x] 독립 source 3건 이상 + official/source diversity + contradiction 낮음 → `supported`
- [x] official 1건 이상 + 독립 source 4건 이상 + forecast confidence medium 이상 → `strong`

출력 문장:

- `blocked`: "외부 수집 승인 전이라 결론을 내릴 수 없습니다."
- `insufficient_evidence`: "현재 수집 자료로는 판정 보류가 맞습니다."
- `provisional`: "현재 잠정 판단은 X입니다."
- `supported`: "현재 판단은 X입니다."
- `strong`: "높은 신뢰도로 X라고 판단합니다."

회귀 기준:

- [x] 무관한 뉴스만 수집되면 "판정 보류" 출력
- [x] 관련 source가 부족하면 "현재 판단은 X"라고 단정하지 않음
- [x] 관련 source가 충분하면 `현재 판단` 문장이 맨 위에 출력

## P33. 신뢰도 개선 공정

- [x] 외부 승인 대기

- [x] SourceVet 독립성 부족
- [x] HTML boilerplate/기사 전문 추출 실패
- [x] ACH survivor 다수
- [x] claim graph contradiction 존재
- [x] forecast reference class 불충분
- [x] RAG 근거가 local fixture에 치우침

신뢰도 상승 조건:

- [x] 질문 관련 직접 source 2건 이상
- [x] 서로 독립된 출처 2개 이상
- [x] 공식 source 또는 전문 source 1개 이상
- [x] SourceVet pass
- [x] ACH survivor 1~2개로 축소
- [x] claim graph contradiction 0 또는 설명 완료
- [x] forecast confidence medium 이상
- [x] article extraction이 title/body/date/source를 안정적으로 분리

구현 항목:

- [x] `ConfidenceAssessment` 타입 추가 (`RuntimeConfidenceAssessment`)
- [x] confidence blockers/reducers/promoters 분리
- [x] report에 `신뢰도 개선 방법` 섹션 추가
- [x] low confidence일 때 "다음 수집 목표"를 자동 생성
- [x] 질문 유형별 수집 목표 template 추가

사용자 출력 예:

```text
신뢰도: 낮음
낮은 이유:
- 직접 관련 미국 공식 반응 source가 없음
- 수집된 뉴스 5건 중 4건은 질문 관련도 낮음

신뢰도를 올리려면:
- 미국 국무부/백악관/의회 발언 source 1건 이상 확인
- 주요 외신 또는 지역 전문 매체 2건 이상 교차 확인

```

## 9. 브리핑

현재 WARDEN은 기능적으로는 이미 "돌아가는 에이전트 런타임"에 가까워졌다. 문제는 분석 결과를 사용자에게 보여주는 product surface가 아직 내부 자료구조를 너무 많이 노출한다는 점이다.

가장 먼저 고쳐야 할 것은 **기사 전문 미출력과 중복 보고서 제거**다. 이 두 가지를 고치면 터미널 경험이 크게 개선된다. 그 다음은 **Runtime Verdict Engine**이다. ACH의 "생존 가설"은 내부 분석 상태이고, 사용자가 원하는 것은 "그래서 현재 판단이 무엇인가"다.

신뢰도 문제는 단순히 문장을 더 세게 쓰는 것으로 해결하면 안 된다. 오히려 관련 source 수, 독립성, SourceVet 통과, ACH survivor 축소, forecast confidence를 조합해 "판정 보류/잠정 판단/강한 판단"으로 나누는 편이 맞다. 그러면 WARDEN은 확정적으로 보이되, 근거가 부족할 때는 정확히 멈출 수 있다.

이 개선이 끝나면 WARDEN CLI는 "긴 내부 로그를 뿌리는 도구"에서 "단계별 분석 진행과 최종 판단을 보여주는 전략분석 콘솔"로 올라간다.
