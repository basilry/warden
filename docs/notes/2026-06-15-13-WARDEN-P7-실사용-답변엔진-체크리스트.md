# WARDEN P7 실사용 답변 엔진 체크리스트

## 왜 P7이 필요한가

2026-06-15 실사용 테스트에서 `warden` CLI는 정상 실행됐지만, 사용자가 기대한 답변은 나오지 않았다.

입력:

```text
대한민국 및 동북아 공급망에 대해 알려줘
```

출력 성격:

- runtime loop 진행 로그
- model proposal 수신 로그
- tool call 성공/차단 로그
- ACH 생존 가설 요약
- approval queue

부족한 점:

- 질문에 대한 직접 답변이 없다.
- 현재 근거로 말할 수 있는 내용과 승인 후 보강할 내용이 분리되지 않는다.
- Codex가 생성한 proposal이 사용자-facing answer로 변환되지 않는다.
- `run_warden_team` 결과가 brief/answer로 렌더링되지 않고 요약 필드만 출력된다.

P7 목표:

> WARDEN을 “루프가 도는 CLI”에서 “검증된 범위 안에서 답변하는 CLI”로 올린다.

## 범위

포함:

- `RuntimeAnswer` 타입
- deterministic answer composer
- optional model-assisted answer composer
- answer authority validator
- CLI answer panel
- waiting approval 상태에서 partial answer 출력
- answer regression

제외:

- 외부 OSINT 실제 수집
- 승인 후 resume
- 웹 콘솔
- 다중 사용자 권한
- 운영용 RAG/검색 인덱스

## P7.0 Answer Contract

### 목표

사용자-facing answer를 runtime output의 1급 산출물로 만든다.

### 생성 파일

| 파일 | 목적 |
|---|---|
| `src/runtime/answer.ts` | answer 타입과 validator 입력 타입 |
| `src/runtime/answer-composer.ts` | deterministic/model-assisted composer |

### 수정 파일

| 파일 | 수정 내용 |
|---|---|
| `src/runtime/types.ts` | `outputs.answer?: RuntimeAnswer` 추가 |
| `src/runtime/loop.ts` | tool 결과 이후 answer 생성 |

### 핵심 타입

```ts
type RuntimeAnswer = {
  title: string;
  directAnswer: string;
  keyFindings: string[];
  evidenceUsed: string[];
  uncertainty: string[];
  blockedActions: string[];
  nextSteps: string[];
  authorityRefs: string[];
};

type AnswerContext = {
  objective: string;
  runStatus: RuntimeRun["status"];
  teamResult?: TeamRunResult;
  approvals: ApprovalRequest[];
  modelResponses: ModelResponse[];
};
```

### 체크리스트

- [x] answer에 `authorityRefs` 포함

## P7.1 Deterministic Answer Composer

### 목표

LLM이 없어도 현재 ACH/SourceVet/approval 결과만으로 최소 답변을 만든다.

### 핵심 함수

- `buildAnswerContext(run, teamResult): AnswerContext`
- `composeDeterministicAnswer(context): RuntimeAnswer`
- `renderFindingFromAch(result): string[]`
- `renderUncertaintyFromVerification(result): string[]`
- `renderBlockedActions(approvals): string[]`

### 로직

1. 사용자 objective를 title과 direct answer의 기준 질문으로 둔다.
2. ACH survivors를 `keyFindings`로 변환한다.
3. SourceVet flags와 verifier residual risk를 `uncertainty`로 변환한다.
4. pending approval을 `blockedActions`로 변환한다.
5. RFI를 `nextSteps`로 변환한다.
6. fixture 기반 답변이면 “현재 로컬/fixture 기반 분석”이라고 명시한다.

### 체크리스트

- [x] ACH survivors -> key findings
- [x] ACH ranking -> 요약 판단
- [x] RFI -> next steps
- [x] SourceVet flags -> uncertainty
- [x] verifier residual risk -> uncertainty
- [x] pending approval -> blocked actions
- [x] fixture/local limitation label
- [x] 한국어 답변 문장 템플릿

완료 기준:

- mock/codex 여부와 관계없이 항상 최소 답변이 나온다.

## P7.2 Model-Assisted Answer Composer

### 목표

Codex/LLM은 검증된 context를 바탕으로 문장 품질만 개선한다. 권위값은 수정하지 못한다.

### 핵심 함수

- `composeModelAssistedAnswer(context, model): Promise<RuntimeAnswer>`
- `buildAnswerDraftPrompt(context): ModelRequest`
- `parseAnswerDraft(response): RuntimeAnswerDraft`
- `mergeDraftWithAuthorities(draft, deterministic): RuntimeAnswer`

### 로직

1. deterministic answer를 먼저 만든다.
2. 모델에는 deterministic answer, ACH/SourceVet/approval 요약만 제공한다.
3. 모델은 `directAnswer`와 표현 개선만 제안한다.
4. `keyFindings`, `uncertainty`, `blockedActions`, `authorityRefs`는 deterministic 값을 우선한다.
5. 모델 출력이 JSON parse에 실패하면 deterministic answer로 fallback한다.

### 체크리스트

- [x] answer draft prompt 작성
- [x] JSON response schema 정의
- [x] invalid JSON fallback
- [x] model timeout/error fallback
- [x] deterministic authority 우선 merge
- [x] no ungrounded claim validator와 연결

완료 기준:

- Codex가 느리거나 실패해도 답변은 나온다.
- Codex가 새 사실을 단정하면 validator가 막거나 “검증 필요”로 내린다.

### 출력 예시

```text
답변
현재 로컬 근거 기준으로는 대한민국 및 동북아 공급망 리스크를 특정 단일 원인으로 단정할 수 없습니다.
다만 WARDEN의 통제 분석에서는 제재 우회 비축과 공급망 교란 가설이 현재까지 생존했습니다.

핵심 판단
- 제재 우회 비축: 현재 ACH 생존 가설
- 공급망 교란: 현재 ACH 생존 가설

한계
- 외부 OSINT 수집은 승인 전이라 실행되지 않았습니다.
- 현재 답변은 로컬 fixture/ingestion 근거에 제한됩니다.

다음 단계
- external_osint_fetch 승인 후 공개 출처를 수집하고 SourceVet으로 재검토
```
